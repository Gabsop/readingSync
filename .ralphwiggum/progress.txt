### Completed (continued 2)
- [x] Sync engine: push/pull with ReadingSync API
  - New `sync-engine.ts` module (`mobile/lib/sync-engine.ts`) with full push/pull sync
  - **Device identity**: generates and persists a unique device ID via `expo-secure-store`
  - **Push sync (enqueue + flush)**:
    - `enqueueSync()`: inserts progress into `sync_queue` table, coalesces pending entries per book
    - Debounced flush (3 seconds) — batches rapid page turns into one API call
    - `flushSyncQueue()`: processes pending queue items, POSTs to `POST /api/progress`
    - Handles "skipped" responses (server has newer timestamp) — marks as synced, no retry
    - Exponential retry on network errors, max 5 attempts before marking as failed
    - Auto-cleanup of old synced/failed entries (24h retention)
  - **Pull sync (on book open)**:
    - `fetchRemoteProgress()`: GETs latest progress from `GET /api/progress/:bookId`
    - `resolveProgressOnOpen()`: compares local vs remote, handles conflict resolution logic
    - Same-device detection: skips if remote is our own echo (device ID match)
    - Staleness threshold: ignores remote positions older than 7 days
    - Silent apply: if remote is <5% different from local, applies without prompting
    - Prompt action: if >5% different, returns "prompt" for future conflict UI
  - **Reader integration**:
    - `saveProgressToDb()` now enqueues sync after each local save
    - Non-blocking remote progress fetch on book open (background)
    - User navigation suppresses late-arriving sync prompts (`userHasNavigatedRef`)
    - Sync queue flushed on reader unmount (fire-and-forget)
    - `cancelPendingFlush()` called on cleanup to prevent stale timers
  - Offline-first: all reads work without network, syncs when connectivity returns
  - TypeScript passes cleanly, Expo build succeeds

### Completed (continued 3)
- [x] Excerpt-based text matching for Kindle sync
  - New `extractPlainText(epub, spineIndex)` function in `epub-parser.ts`: strips XHTML tags,
    decodes HTML entities, normalizes whitespace to produce searchable plain text
  - New `searchExcerpt(epub, excerpt, progressHint?)`: searches all spine chapters for an excerpt
    string, returns matches sorted by proximity to expected global progress position
  - New `extractExcerptAtPosition(epub, spineIndex, positionRatio)`: extracts ~500 chars of text
    at an approximate position within a chapter for cross-device sync payloads
  - Reader now populates `excerpt` field on every progress save — excerpt is stored in SQLite
    `reading_positions` table and included in sync pushes to the backend
  - Remote progress resolution now uses 3-tier strategy:
    1. Same-source mobile positions: parse `{ chapter, page }` JSON directly (existing behavior)
    2. Cross-renderer positions (Kindle): search EPUB for remote excerpt text, estimate page from
       character offset ratio — bypasses incompatible position formats entirely
    3. Percentage fallback: if excerpt not found, approximate position using `progress * spineLength`
  - Excerpt search normalizes whitespace on both sides for robust matching across renderers
  - Search is async and respects `userHasNavigatedRef` — late results are suppressed if user moved on
  - TypeScript passes cleanly

### M3 — Progress sync

#### Completed
- [x] Port sync-engine.ts to SyncCore (push/pull, 409 deferral, device-id)
  - New `SyncEngine` class (`SyncCore/SyncEngine.swift`): @MainActor @Observable, ports all
    sync logic from RN `sync-engine.ts` to native Swift
  - **Device ID**: generated as 16-byte random hex, persisted in Keychain via `KeychainStore`,
    cached in memory — same format as RN `getDeviceId()`
  - **Push sync (enqueue + flush)**:
    - `enqueueSync(_:)`: coalesces pending/deferred entries for same book, inserts into
      `sync_queue` GRDB table, schedules debounced flush (3s via Task.sleep)
    - `flush()`: processes pending queue items, POSTs to `/api/progress` via `APIClient`,
      handles 409 as "deferred", retries up to 5 times with exponential backoff, cleans
      old entries (24h retention)
    - `cancelPendingFlush()`: cancels debounce task on unmount
  - **Pull sync**:
    - `fetchRemoteProgress(bookId:)`: GETs `/api/progress/:bookId`, decodes `RemoteProgress`
    - `resolveProgressOnOpen(bookId:)`: same-device detection, staleness threshold (7 days),
      <5% diff auto-apply, >5% diff prompt — exact port of RN logic
  - **New types** in `Models.swift`:
    - `RemoteProgress` (Codable): backend progress response with snake_case CodingKeys
    - `SyncPayload`: input struct for enqueueSync
    - `SyncProgressBody` (Encodable): POST request body with snake_case CodingKeys
    - `SyncResolution` enum: useLocal / useRemote / prompt
  - **APIClient additions**: `syncProgress(_:)` returns HTTP status code (200/409),
    `fetchBookProgress(bookId:)` decodes RemoteProgress, new `performPost` static helper
  - **Persistence additions**:
    - `SyncQueueRecord` (GRDB FetchableRecord + PersistableRecord) with auto-increment id
    - `v2-sync-queue` migration: creates sync_queue table matching RN schema
    - CRUD methods: `coalesceSyncQueue`, `insertSyncQueueItem`, `pendingSyncQueueItems`,
      `updateSyncQueueStatus`, `cleanOldSyncQueueItems`
  - **SyncCore depends on Persistence** (added to Package.swift) for database access
  - Build: zero errors, zero warnings on Xcode 26 / iOS 26.4 Simulator
  - Tests: 7/7 SyncCore BookIdentity tests still passing

#### Completed (continued)
- [x] Map Readium Locator ↔ EPUB CFI for position field
  - New `LocatorMapper.swift` in EpubReader package: bidirectional conversion between
    Readium `Locator` and compact CFI-based JSON position string
  - `CFIPosition` struct: stores `href`, `type` (media type), `cfi` (EPUB CFI fragment),
    `progression` (intra-resource), and `totalProgression` (whole-book progress)
  - `toCFIString(_:)`: extracts CFI fragment from `Locator.locations.fragments`, serializes
    compact JSON (~100 bytes vs ~300+ for full Locator JSON)
  - `toLocator(_:)`: reconstructs `Locator` from compact JSON with proper `AnyURL` href
    and `MediaType`; falls back to full Readium Locator JSON for backward compatibility
  - `ReaderViewModel.savePosition()` now stores compact CFI format via `LocatorMapper`
  - `ReaderViewModel.savedLocator` restores via `LocatorMapper.toLocator()` with fallback
  - Position format is cross-client compatible: KOReader can produce matching CFI + href
  - Build: zero errors, zero warnings on Xcode 26 / iOS 26.4 Simulator
  - Tests: 7/7 SyncCore BookIdentity tests still passing
